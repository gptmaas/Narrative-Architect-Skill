# Narrative Architect Skill

[English](README.md) | **简体中文**

把复杂的想法，变成有故事感的表达。

大多数稿子都以同一种安静的方式失败：事实都对，但没人感受到它为什么重要。

Narrative Architect Skill 帮你在复杂的素材里找到压力、冲突和转折，再为你真正需要打动的那个人打磨表达。它不会让所有人都喜欢你的话。它让对的听众理解这件事为什么重要。

## 这是什么

这个 skill 像搭故事一样搭沟通：从利害和转折出发，而不是堆要点和套话。

它擅长几件事：

- **找到叙事内核** —— 什么处在风险中，哪个旧假设正在被打破
- **读懂受众** —— 他们知道什么、害怕什么、从哪里切入
- **控制节奏** —— 什么时候让读者安心，什么时候给他们惊喜
- **让事实判断停留在已知边界之内**

对于有素材依据的工作，事实判断始终扣住已知材料。如果素材有缺口，输出会把缺口标出来，而不是用猜测去填上。

## 适合谁

适合那些心里清楚自己想说什么、却总看着别人误解的人。

对准备路演的创始人、解释架构的工程师、写不那么像模板的开发信的 BD 负责人，以及想要结构又不想丢掉自己声音的写作者，都有用。

## 试试看

- "写一篇文章，讲为什么'数据割裂'比'数据缺失'更危险。"
- "帮我通过朋友介绍给一位专家发消息，我想探讨合作。"
- "把这个技术架构讲给没有工程背景的人听。"
- "明天的路演之后，给我一段简短的跟进。"
- "不是冷启动开发信，是朋友介绍的。"

## 它能帮你做什么

| 任务 | 例子 |
| --- | --- |
| 公开叙事 | 把一次产品发布，变成有张力、有筹码、有让人记住的收尾的故事 |
| 一对一策略 | 给投资人、专家或合作伙伴写一条能推进关系的消息 |
| 专家转译 | 把一个技术概念讲给另一类受众，而不降低它的层次 |
| 短消息 | 用几句话起草一封温暖的引荐、跟进或会议确认 |
| 个人沟通 | 起草一条个人、情感或敏感话题的消息，且保持自然 |
| 重写修复 | 诊断一份稿子为什么没打动人，然后换条路重写，而不是表面润色 |

## 它怎么工作

这个 skill 会选择能完成任务的最轻量模式。短消息不会被吹成提案。一次纠正不会只做表面润色。

| 模式 | 什么时候用 |
| --- | --- |
| Quick Message（短消息） | 简短留言、微信、跟进 |
| One-on-One Strategy（一对一策略） | 联系某个人、团队、投资人、专家或合作伙伴 |
| Public Narrative（公开叙事） | 文章、演讲、品牌故事、社交平台帖子 |
| Expert Translation（专家转译） | 把一个概念讲给另一类受众 |
| Rewrite Recovery（重写修复） | 修复一份没打动人的稿子 |
| Personal Communication（个人沟通） | 个人、情感、友谊或敏感话题的消息 |
| Full Narrative（完整叙事） | 重大路演、宣言、主题演讲、长文 |

## 原则

**讲故事，而不是推销。**
目标不是让所有人喜欢。是与对的听众产生强共鸣。

**有控制的意外。**
好的沟通在"我懂了"和"我没从这个角度看过"之间来回移动。太可预测会无聊，太混乱会读不下去。这个 skill 在管理这种节奏。

**受众优先，权威保留。**
受众决定切入点与节奏。你的中心判断不变。

**事实边界。**
对于有素材依据的工作，事实判断始终扣住已知材料。如果素材有缺口，输出会把缺口标出来，而不是用猜测去填上。

## 安装

克隆仓库：

```bash
git clone https://github.com/gptmaas/Narrative-Architect-Skill.git
```

常见的本地做法，是把 skill 目录放在你 OpenClaw 工作区的 `skills/` 下：

```
skills/narrative_architect_skill/
```

具体依赖取决于你的 OpenClaw 运行时与模型提供方配置。

## 项目结构

仓库的简化结构如下：

```
Narrative-Architect-Skill/
├── LICENSE
└── narrative_architect_skill/
    ├── skill.md              # 规则与原则
    ├── runtime_chain.md      # 运行链与模式
    ├── mappers/              # 故事结构、受众、信息熵
    ├── principles/           # 核心叙事原则
    ├── selectors/            # 运行模式与切入点选择
    ├── renderers/            # 渲染器与风格封装
    ├── harness/              # 质量与校验检查
    ├── contracts/            # 输出格式契约
    ├── policies/             # 行为策略
    └── cases/                # 回归测试
```
