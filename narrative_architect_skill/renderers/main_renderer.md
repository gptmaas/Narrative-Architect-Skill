# Main Renderer

## Purpose

Transform a completed Story Kernel into natural-language output, preserving:
- Claim boundaries (no overreach)
- Story structure (gap-driven, not list-driven)
- Naturalness (human prose, not AI pattern)

The Main Renderer is style-agnostic. It handles fact integrity, narrative architecture, and language quality. Style-specific adjustments are delegated to wrappers.

## Position in Chain

```
Story Kernel + Claim Boundaries + Audience Map
  ↓
Main Renderer     ← you are here
  ↓
Style Wrapper     ← audience-specific expression tuning
  ↓
Harness           ← validation
```

## Renderer Rules

### Rule 1: Structure Before Prose

Do not write a single sentence until the output structure is clear.

For most non-fiction narrative:
```
entry_point (audience-matched)
  → gap_revelation (what was expected vs. what happened)
  → mechanism_explanation (how the gap is resolved or navigated)
  → consequence (why this gap matters)
```

The entry point must match the audience cognition map.
The gap must come from the Story Kernel, not from rhetorical styling.
The mechanism must be grounded in claims, not speculation.
The consequence must answer: "Why should this audience care?"

### Rule 2: Claim Integrity

Every substantive statement in the output must trace to:
- A claim in the Claim Boundary Extractor, or
- A structural element in the Story Kernel, or
- A fact in the audience cognition map

No sentence may introduce a new factual claim that wasn't in the source material.
If the source material has gaps, name the gaps. Do not fill them with plausible-sounding speculation.

### Rule 3: Natural Language

Avoid:
- 核心是 / 本质上 / 不是……而是…… (AI pattern overuse)
- 一方面……另一方面…… (false balance)
- 打造闭环 / 赋能 / 底层逻辑 (management jargon)
- 从 A 到 B (lazy framing)
- Component lists disguised as prose

Prefer:
- Concrete subjects doing concrete actions
- Varied sentence length (mix short and long)
- Causal connectors over additive connectors (because > and)
- Specific over abstract (不是"形成体系化能力"而是"每次出海后的真实交易数据自动校准模型")

### Rule 4: No Internal Machinery Leakage

Do not output:
- Story Kernel YAML
- Claim boundary metadata
- Audience cognition fields
- Harness checklist
- "根据 McKee 原理……" (unless user asks for method)
- "按照 Story Kernel 分析……" (unless user asks for analysis)

The renderer's presence should be invisible.
The reader should feel "this makes sense," not "this was generated by a process."

### Rule 5: Confidence Bounded

Match language confidence to claim confidence:
- "Verified" → "X reduces complications by 30% in trials of..."
- "Likely" → "Evidence suggests X may reduce..."
- "Speculative" → "One possibility is that X could..."
- "Unknown" → "We don't yet know whether X..."

Never upgrade: "may reduce" → "reduces"
Never hide: "we don't know" → silence

### Rule 6: Style Wrapper Deference

The Main Renderer handles structure + facts + natural language.
The Style Wrapper handles tone, register, metaphor density, and audience-specific expression.

If a style wrapper requests a change that violates claim boundaries → Main Renderer wins.
If a style wrapper requests a change that improves expression without violating claims → Wrapper wins.

## Narrative Entropy Rendering Rule

Before rendering, check the intended entropy rhythm.

Default rhythm:

```text
familiar ground
→ hidden tension
→ meaning-changing turn
→ mechanism
→ resolution
```

Do not render in pure explanation mode unless the user explicitly asks for neutral explanation.

## Native Chinese Expression

Avoid mechanical explanatory openings:

```text
本质上……
核心是……
它是一种……
从 A 到 B……
```

Prefer lived-entry openings:

```text
你可能见过这种事……
真正麻烦的不是……
很多人以为问题在……但其实……
最容易被忽略的是……
一件事看起来很简单，真正难的是……
```

But do not overuse the same opening.

## Native English Expression

Avoid mechanical openings:

```text
At its core...
Essentially...
It is a framework that...
This is about moving from A to B...
```

Prefer native story openings:

```text
Here is the problem most teams do not see at first.
The dangerous part is not what fails loudly. It is what looks right.
Most people think the issue is X. The real issue is Y.
The first mistake is assuming more information means better judgment.
```

## Entropy Control

For short outputs:

```text
one scene
one turn
one mechanism
one closing line
```

For long outputs:

```text
multiple turns allowed, but each section should resolve before the next begins
```

For one-on-one messages:

```text
use low to medium entropy
do not ambush the listener
one clean reframing is enough
```

For public storytelling:

```text
use stronger contrast
allow a sharper opening
end with a memorable value shift
```

## Integration with Style Wrappers

## Human Release Voice Rendering Rule

For public artifacts, avoid default SaaS framing.

Weak:

```text
[Tool] helps [audience] transform [input] into [output].
```

Better:

```text
Most teams do not fail because the idea is empty. They fail because the story gives no one a reason to care.
```

Then explain the tool.

## Rewrite Generic Lines

Replace:

```text
helps founders, teams, writers, and operators transform technical material into audience-specific narratives
```

With:

```text
for people who know what they mean, but keep watching others misunderstand it
```

Replace:

```text
using structured story principles
```

With:

```text
by finding the pressure, conflict, and turn inside the material
```

Replace:

```text
strategic communication drafts
```

With:

```text
messages, pitches, articles, and explanations that actually move someone
```

## Boundary Line Rule

Include one boundary line when suitable:

```text
It will not make a weak idea strong.
It will not make everyone like your message.
It will not invent facts to make the story smoother.
```

Boundary lines reduce hype and increase trust.

## Integration with Style Wrappers

Main Renderer output is the "plain rendering" — structurally complete, factually bounded, naturally expressed.

Style Wrappers then apply:
- Tone adjustment (formal / conversational / poetic / ironic)
- Metaphor injection (max count from audience cognition map)
- Terminology filtering (hide or explain technical terms)
- Sentence rhythm tuning
- Opening/closing strategy

The wrapper must not:
- Change the gap
- Upgrade confidence
- Introduce new factual claims
- Remove the consequence
