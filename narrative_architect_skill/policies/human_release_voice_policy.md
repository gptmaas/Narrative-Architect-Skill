# Human Release Voice Policy

## Purpose

This policy reduces AI-like surface patterns in public artifacts.

It applies to:

```text
GitHub README
open-source documentation
launch posts
landing pages
marketplace listings
public product pages
public skill descriptions
developer-facing docs
```

## Core Rule

Human voice is not casualness.

Human voice means:

```text
specific judgment
visible tradeoff
real use case
uneven rhythm
bounded confidence
less template symmetry
```

AI-like writing often sounds:

```text
complete
balanced
polished
generic
evenly structured
full of abstract nouns
```

Human writing often has:

```text
a reason for existing
a point of view
concrete situations
some compression
some asymmetry
one or two sharp lines
```

## AI-Smell Patterns

Watch for:

```text
helps X transform Y into Z
whether you're A, B, or C
seamlessly
empower
unlock
leverage
robust
powerful
comprehensive
end-to-end
strategic communication drafts
audience-specific narratives
core principles
mode-aware processing
```

These words are not banned. But repeated use creates template smell.

## Humanizing Rules

### 1. Replace broad audience lists with sharper user situations

Weak:

```text
For founders, teams, writers, and operators.
```

Better:

```text
For people who know what they mean, but keep watching others misunderstand it.
```

### 2. Replace abstract value with lived friction

Weak:

```text
Transform technical material into audience-specific narratives.
```

Better:

```text
Turn the thing in your head into something another person can actually follow, feel, and act on.
```

### 3. Add one honest boundary

Human copy is more believable when it says what the tool is not.

Examples:

```text
It will not make a weak idea strong.
It will not replace your judgment.
It will not make everyone like your message.
It helps you find the story that the right person can enter.
```

### 4. Make one sentence carry judgment

Good public copy should contain at least one sentence that could only come from a point of view.

Examples:

```text
Most bad communication is not unclear. It is uncommitted.
The problem is rarely that the audience is stupid. The problem is that the story has no pressure.
A message that tries to please everyone usually gives no one a reason to care.
```

### 5. Use fewer symmetrical lists

Lists are useful, but too many lists make public artifacts feel generated.

For README, prefer:

```text
short list for features
short list for examples
short list for install
```

Avoid listing every internal capability in the first half.

### 6. Vary rhythm

Mix:

```text
short sentence
medium explanation
one concrete example
short closing line
```

Avoid paragraph after paragraph of same-length explanatory prose.

### 7. Keep technical structure, but move it below the fold

Internal architecture is useful, but human readers need the problem first.

Public order:

```text
problem
value
examples
usage
then architecture
```

## README Human Voice Rule

For public README, add or preserve:

```text
one felt problem
one clear boundary
one concrete input example
one non-generic closing line
```

## Failure Modes

Hard fail:

1. The README is correct but sounds like SaaS landing-page filler.
2. The first screen has no lived problem.
3. The copy lists audiences but does not name a real pain.
4. The artifact has no point of view.
5. Every section is equally polished and equally bland.

Soft fail:

1. Too many abstract nouns.
2. Too many parallel lists.
3. Too many "helps X do Y" sentences.
4. The language is clean but forgettable.
5. The tool sounds useful but not necessary.

## Rewrite Rule

If the artifact has AI smell, rewrite using:

```text
felt problem
→ point of view
→ what the tool does
→ concrete examples
→ honest boundary
→ practical usage
```

## Final Rule

Do not make the artifact sound more casual.
Make it sound more owned.
